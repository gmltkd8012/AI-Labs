# AI 엔지니어링 변천사: 프롬프트 → 컨텍스트 → 하네스 → 에이전틱

사용자의 의도를 AI가 정확히 이해하고 실행하기 위해 발전해온 기술의 진화 과정

---

## 1️⃣ Prompt Engineering (프롬프트 엔지니어링)

### 📖 정의
인공지능이 사용자의 의도를 정확히 이해하고 최적의 결과물을 출력할 수 있도록 명령어와 맥락을 구성하는 기술

### 🎯 핵심 개념
**"올바른 질문을 올바르게 던지기"**

AI의 응답 품질은 입력(프롬프트)의 질에 100% 의존합니다. 프롬프트 엔지니어링은 이 입력을 최적화하여 원하는 출력을 얻는 기술입니다.

### 🔑 4가지 핵심 요소

#### 1. 명령 (Instruction)
- **역할**: AI가 수행해야 할 구체적인 작업 내용
- **예시**: 
  ```
  ❌ "코드 작성해"
  ✅ "Python으로 사용자 로그인 기능을 구현하고, 
       예외 처리와 주석을 포함해줘"
  ```

#### 2. 맥락 (Context)
- **역할**: AI가 답변을 구성하는데 필요한 배경 정보
- **예시**:
  ```
  "너는 5년 경력의 백엔드 개발자야.
   Spring Boot와 MySQL을 주로 사용했어.
   보안을 최우선으로 고려해야 해."
  ```

#### 3. 입력 데이터 (Input Data)
- **역할**: 답변을 얻고자 하는 질문이나 대상
- **예시**:
  ```
  "다음 코드의 버그를 찾아줘: [코드 붙여넣기]"
  "이 JSON 구조를 Kotlin 데이터 클래스로 변환해줘"
  ```

#### 4. 출력 형식 (Output Format)
- **역할**: 답변의 형태를 명확하게 지정
- **예시**:
  ```
  "JSON 형식으로 다음 정보를 정리해줘"
  "마크다운 테이블로 비교 분석해줘"
  "코드와 설명을 함께 제공해줘"
  ```

### 📋 예시: 완성된 프롬프트

```
당신은 Android 개발 전문가입니다.

명령: Kotlin으로 RecyclerView 어댑터를 작성하세요.

맥락: 
- API에서 받은 사용자 목록을 표시할 것입니다
- 각 항목은 프로필 이미지, 이름, 이메일을 포함합니다
- ViewBinding을 사용합니다

입력 데이터:
data class User(
    val id: Int,
    val name: String,
    val email: String,
    val profileImageUrl: String
)

출력 형식:
- 완전한 코드 작성
- 주요 함수에 주석 추가
- MVVM 패턴을 따를 것
```

### 장점 & 단점

| 구분 | 내용 |
|------|------|
| ✅ **장점** | - 구현 간단<br>- 빠른 결과<br>- 기초 학습 용이 |
| ⚠️ **한계** | - 실시간 정보 불가<br>- 복잡한 작업 어려움<br>- 컨텍스트 길이 제한<br>- 상태 관리 불가 |

## 2️⃣ Context Engineering (컨텍스트 엔지니어링)

### 📖 정의
AI 모델에 질문을 잘 던지는 '프롬프트 엔지니어링'을 넘어, AI가 복잡한 작업을 수행할 수 있도록 필요한 정보와 도구를 적절한 형식과 시점에 맞게 제공하는 동적 시스템을 설계하는 일

### 🎯 핵심 개념
**"프롬프트는 전체 입력 생태계의 일부일 뿐"**

Prompt Engineering이 "어떻게 질문할지"라면, Context Engineering은 "어떤 정보를 주고, 어떤 도구를 제공하고, 어떤 메모리를 유지할지"를 설계합니다.

### 🔑 4가지 핵심 요소

#### 1. 지침 (Instructions/Guidelines)
- **역할**: AI의 역할, 페르소나, 반드시 지켜야 할 규칙
- **예시**:
  ```
  당신은 보안 전문가입니다.
  
  규칙:
  - 절대 민감한 정보(비밀번호, API 키)를 출력하지 않기
  - 항상 보안 위험을 먼저 언급하기
  - 베스트 프랙티스에 따른 조언만 제공하기
  ```

#### 2. 지식 (Knowledge)
- **역할**: RAG 등을 통해 외부에서 끌어온 실시간 최신 정보 및 데이터
- **구성**:
  - 회사 내부 문서 및 코딩 가이드라인
  - 최신 뉴스, API 문서
  - 프로젝트 스펙과 요구사항
  - 사용자 히스토리 및 선호도
  
- **예시** (RAG 활용):
  ```
  사용자: "우리 회사 API 문서에 따르면 
           이 요청은 어떻게 처리해야 해?"
  
  시스템:
  1. 회사 API 문서 검색 (RAG)
  2. 해당 엔드포인트 스펙 추출
  3. AI가 그 정보를 바탕으로 답변
  ```

#### 3. 도구 (Tools)
- **역할**: AI가 실행할 수 있는 계산기, 검색 엔진, 외부 API 등
- **예시**:
  ```
  - calculator (수학 계산)
  - web_search (실시간 검색)
  - code_executor (코드 실행 및 테스트)
  - database_query (데이터베이스 조회)
  - send_email (이메일 전송)
  - file_system (파일 읽기/쓰기)
  ```

#### 4. 기억 및 대화 이력 (Memory & Conversation History)
- **역할**: 이전 상호작용 및 세션 데이터를 유지
- **종류**:
  - **단기 기억**: 현재 대화의 이전 메시지들
  - **장기 기억**: 사용자의 선호도, 과거 질문, 완료된 작업
  - **요약**: 긴 대화를 압축하여 컨텍스트 유지
  
- **예시**:
  ```
  1차 상호작용:
  사용자: "내 이름은 김철수야, Python으로 학습중이야"
  
  2차 상호작용 (며칠 후):
  사용자: "지난번 배운 내용 복습해줄 수 있어?"
  시스템: (기억 활용) "네, 당신이 Python 초급이라는 걸 
           알고 있으니, 함수 개념부터 시작할게"
  ```

### 🔄 Prompt Engineering vs Context Engineering

| 구분 | Prompt Engineering | Context Engineering |
|------|-----------------|-----------------|
| **범위** | 질문 하나의 최적화 | 전체 시스템 설계 |
| **초점** | 명령어 문구 | 정보, 도구, 메모리, 규칙 |
| **복잡도** | 낮음 | 높음 |
| **비용** | 낮음 | 중간 |
| **효율성** | 제한적 | 높음 |
| **실시간 정보** | 불가능 | 가능 (RAG 활용) |
| **멀티스텝 작업** | 어려움 | 가능 |

### 장점 & 단점

| 구분 | 내용 |
|------|------|
| ✅ **장점** | - 실시간 정보 활용 가능<br>- 복잡한 멀티스텝 작업 지원<br>- 사용자별 맞춤화<br>- 토큰 효율성 향상 |
| ⚠️ **한계** | - 복잡한 시스템 구축 필요<br>- 운영 오버헤드 증가<br>- 장기 자율 작업은 여전히 어려움 |

## 3️⃣ Harness Engineering (하네스 엔지니어링)

### 📖 정의
AI 코딩 에이전트가 복잡하고 장기적인 작업을 수행할 때, 궤도를 이탈하거나 실수를 반복하지 않도록 통제하고 피드백 루프를 제공하는 전체 실행 환경 설계 기술

### 🎯 핵심 개념
**"말(LLM)은 강력하지만, 마구(Harness)를 씌워야 농사(실제 일)를 할 수 있다"**

Context Engineering이 정보와 도구를 제공한다면, Harness Engineering은 AI가 그것들을 안전하고 효율적으로 **사용하도록 감시하고 통제하는** 시스템입니다.

### 🔑 5가지 핵심 요소

#### 1. 컨텍스트 (Context)
- **역할**: 에이전트가 문맥을 잃지 않고 작업할 수 있도록 돕는 구조화된 프로젝트 문서 및 지침
- **구성**:
  - `AGENTS.md`: 에이전트가 따라야 할 지침서
  - 프로젝트 구조 문서
  - 코딩 표준 및 규칙
  - API 문서
  
- **예시** (AGENTS.md):
  ```markdown
  # AGENTS.md - 코딩 에이전트 가이드
  
  ## 역할
  - Kotlin Android 앱 개발
  - MVVM 아키텍처 준수
  
  ## 금지사항
  - deprecated API 사용 금지
  - 민감한 정보를 로그에 남기지 말 것
  - 동기 네트워크 호출 금지
  
  ## 디렉토리 구조
  - src/main/kotlin/presentation/ - UI 레이어
  - src/main/kotlin/domain/ - 비즈니스 로직
  - src/main/kotlin/data/ - 데이터 레이어
  ```

#### 2. 도구 (Tools)
- **역할**: 에이전트가 외부 세계와 상호작용하고 작업을 수행할 수 있게 하는 API, 스킬, 도구 등
- **예시**:
  ```
  - 파일 시스템 접근 (읽기/쓰기)
  - 코드 컴파일 및 실행
  - 테스트 프레임워크 (JUnit, Espresso)
  - 버전 관리 (Git)
  - 브라우저 (Web 검색)
  - 터미널 명령어 실행
  - 데이터베이스 쿼리
  ```

#### 3. 가드레일 및 검증 (Guards & Verification)
- **역할**: AI가 규격을 위반하거나 같은 실수를 반복하지 않도록 강제하는 규칙
- **종류**:
  - **정적 검증**: 린터, 타입 체커
  - **자동 테스트**: 단위 테스트, 통합 테스트
  - **규칙 엔진**: "~API는 절대 사용하면 안됨" 같은 규칙
  - **권한 제어**: 어떤 폴더는 수정할 수 없도록 제한
  
- **예시**:
  ```kotlin
  // ❌ AI가 생성한 코드
  val password = "user_password"
  Log.d("AUTH", "Password: $password")
  
  // Linter가 감지: "민감한 정보를 로그에 남기지 말 것"
  // 가드레일 실행: 코드 생성 차단 및 AI에 피드백 전송
  
  // ✅ AI 재시도 - 수정된 코드
  Log.d("AUTH", "Login attempt for user")
  ```

#### 4. 피드백 루프 (Feedback Loop)
- **역할**: 에이전트가 실패 패턴을 분석하고 스스로 수정할 수 있도록 돕는 오류 감지 및 재시도 시스템
- **동작 방식**:
  ```
  1. AI가 코드 작성
     ↓
  2. 테스트 실행
     ↓
  3. 테스트 실패? 오류 메시지 수집
     ↓
  4. 오류 메시지를 AI에게 피드백
     ↓
  5. AI가 코드 수정
     ↓
  6. 반복 (성공할 때까지)
  ```

- **예시** (Kotlin 컴파일 오류):
  ```
  AI: fun getUserName(): String = null
  
  컴파일러: "Type mismatch: inferred type is 
            String? but String was expected"
  
  AI가 받은 피드백으로 수정:
  fun getUserName(): String? = null
  // 또는
  fun getUserName(): String = ""
  ```

#### 5. 상태 관리 (State Management)
- **역할**: 장기 작업 과정에서의 진행 상황, 의존성, 완료된 작업 추적
- **예시**:
  ```
  작업: "로그인 기능 완성"
  
  상태 추적:
  ✅ [완료] 로그인 화면 UI 작성
  ✅ [완료] ViewModel 구현
  🔄 [진행중] API 연동
  ⏳ [대기중] 유닛 테스트
  ⏳ [대기중] 통합 테스트
  ```

### 🔄 Context Engineering vs Harness Engineering

| 구분 | Context Engineering | Harness Engineering |
|------|-----------------|-----------------|
| **초점** | 정보/도구 제공 | 실행 통제 및 감시 |
| **역할** | "AI에게 필요한 정보를 준다" | "AI의 행동을 감시하고 바로잡는다" |
| **검증** | 최소한의 검증 | 강력한 검증 및 가드레일 |
| **피드백** | 직접적 | 자동화된 루프 |
| **복잡도** | 중간 | 높음 |
| **작업 종류** | 일회성 질문 | 장기 자율 작업 |

### 장점 & 단점

| 구분 | 내용 |
|------|------|
| ✅ **장점** | - 장기 자율 작업 가능<br>- 자동 오류 수정<br>- 일관성 있는 출력<br>- 생산성 극대화 |
| ⚠️ **단점** | - 설계 및 구축 복잡<br>- 운영 오버헤드 큼<br>- 초기 투자 비용 높음<br>- 다양한 도메인에 적용 어려움 |

### 📊 실제 성과

OpenAI는 3명의 엔지니어가 하네스 엔지니어링을 통해:
- **100만 줄의 코드** 자동 생성
- **엔지니어당 3.5개의 PR** 제출 (수동 코딩 거의 없음)
- **0개의 수동 작성 코드**

## 4️⃣ Agentic Engineering (에이전틱 엔지니어링)

### 📖 정의
AI 에이전트가 구조화된 인간의 감독 하에 코드를 계획, 작성, 테스트 및 배포하는 시스템을 설계하는 학문

### 🎯 핵심 개념
**"완전 자율 에이전트에 가까운 시스템"**

Harness Engineering이 일반적인 AI의 통제 프레임워크라면, Agentic Engineering은 여러 에이전트가 협력하여 복잡한 소프트웨어 엔지니어링 작업 전체를 수행하는 시스템입니다.

### 🔑 특징

#### 1. Multi-Agent Collaboration (멀티 에이전트 협력)
- **구조**:
  ```
  ┌─────────────┐
  │   Task      │ (사용자 요청)
  │  "기능 X를  │
  │  구현해줘"  │
  └─────┬───────┘
        ↓
  ┌─────────────────────────────────────┐
  │ Agentic Engineering System          │
  ├─────────────────────────────────────┤
  │ ┌─────────────┐  ┌────────────────┐ │
  │ │  Planning   │→ │   Design       │ │
  │ │  Agent      │  │   Agent        │ │
  │ └─────────────┘  └────────────────┘ │
  │        ↓                ↓             │
  │ ┌─────────────┐  ┌────────────────┐ │
  │ │   Coding    │  │    Test        │ │
  │ │   Agent     │  │    Agent       │ │
  │ └─────────────┘  └────────────────┘ │
  │        ↓                ↓             │
  │ ┌─────────────────────────────────┐ │
  │ │    Deployment Agent             │ │
  │ └─────────────────────────────────┘ │
  └─────────────────────────────────────┘
        ↓
  ┌─────────────┐
  │  완성된     │
  │  기능 배포  │
  └─────────────┘
  ```

#### 2. 인간의 감독 (Human Oversight)
- 완전 자동화가 아니라 "구조화된 감독"
- 주요 결정 지점에서 인간 리뷰
- 비상 상황 시 즉시 개입 가능

#### 3. End-to-End Automation (엔드투엔드 자동화)
- 계획 (Planning)
- 설계 (Design)
- 구현 (Implementation)
- 테스트 (Testing)
- 배포 (Deployment)

모든 단계가 자동화되면서도 품질 유지

### 💡 Harness vs Agentic Engineering

| 구분 | Harness Engineering | Agentic Engineering |
|------|-----------------|-----------------|
| **주체** | 단일 AI 에이전트 | 다중 AI 에이전트 |
| **범위** | 특정 작업 (코드 작성) | 전체 소프트웨어 개발 사이클 |
| **복잡도** | 높음 | 매우 높음 |
| **자율성** | 높음 (인간 검증 필요) | 극높음 (최소 감독) |
| **성숙도** | 2026년 초 현재 상용화 중 | 2026년 초 연구 단계 |

### ⏳ 현재 상황 (2026년)

아직 Agentic Engineering에 대한 표준화된 방법론이나 프레임워크가 많지 않습니다. 

**진행 중인 프로젝트들**:
- OpenAI: o1 모델로 복잡한 문제 해결 시스템 개발
- Anthropic: Claude Agent SDK로 멀티 에이전트 프레임워크 구축
- Google: Gemini와 함께 에이전트 협력 시스템 연구
- Meta: LLaMA 기반 오픈소스 에이전트 시스템

**예상 트렌드**:
- 2026년: 에이전트 프레임워크 표준화 시작
- 2027년: 엔터프라이즈 수준의 멀티 에이전트 시스템 등장
- 2028년: 완전 자율 소프트웨어 개발팀 등장 가능성

## 📚 전체 진화 단계 정리

```
사용자의 "의도" 
         ↓
┌─ Prompt Engineering (2023)
│  - 질문을 잘 던지기
│  - 명령, 맥락, 입력, 형식
│  - 한계: 일회성 질문만 가능
│
├─ Context Engineering (2024)
│  - 지침, 지식, 도구, 기억 제공
│  - 동적 정보 시스템 구축
│  - 한계: 장기 자율 작업에는 취약
│
├─ Harness Engineering (2024-2025)
│  - 컨텍스트, 도구, 검증, 피드백, 상태 관리
│  - 자동 오류 수정
│  - 한계: 단일 에이전트의 한계
│
└─ Agentic Engineering (2025-2026)
   - 다중 에이전트 협력
   - 완전 엔드투엔드 자동화
   - 최소 인간 감독으로 전체 개발 사이클 자동화
```

## 🎓 학습 로드맵

### Level 1: 기초 (Prompt Engineering)

- [ ] Chain-of-Thought 기법 습득

## 🚀 각 기술별 실무 적용 예시

### Prompt Engineering 예시
```
작업: 사용자 프로필 데이터 분석

프롬프트:
당신은 데이터 분석가입니다.
다음 사용자 데이터를 분석하여 주요 인사이트 3가지를 찾아주세요.
JSON 형식으로 제공하세요.

[사용자 데이터]
```

### Context Engineering 예시
```
작업: 회사 API 문서에 맞는 기능 구현 제안

시스템:
1. RAG로 최신 API 문서 검색
2. 프로젝트 코딩 규칙 로드
3. 이전 구현 사례 참고 자료 제공
4. AI가 모든 정보를 바탕으로 제안

결과: 회사 규칙과 API를 완벽히 반영한 구현안
```

### Harness Engineering 예시
```
작업: 로그인 기능 개발 자동화

시스템:
1. AI가 로그인 코드 작성
2. 린터 검증 → 스타일 오류 감지
3. 유닛 테스트 자동 실행 → 실패
4. AI에게 오류 피드백 전달
5. AI가 자동 수정
6. 반복 (성공할 때까지)
7. 최종 코드 리뷰 후 병합

결과: 테스트된 고품질 코드 (수동 작성 거의 없음)
```

### Agentic Engineering 예시 (미래)
```
작업: "소셜 미디어 앱의 포스팅 기능 구현"

시스템:
1. Planning Agent: 전체 아키텍처 설계
2. Design Agent: UI/UX 목업 및 API 스펙 정의
3. Backend Agent: 서버 코드 작성
4. Frontend Agent: 모바일 UI 코드 작성
5. Test Agent: 모든 테스트 작성 및 실행
6. Deploy Agent: 스테이징/프로덕션 배포

결과: 완전 자동화된 기능 배포 (인간은 최종 검수만)
```

## 💭 결론

AI 엔지니어링은 단순한 **"좋은 질문 하기"**에서 시작하여, 
**"복잡한 시스템 설계"**를 거쳐, 
**"완전 자율 소프트웨어 개발"**에 이르고 있습니다.

각 단계는 이전 단계를 포함하면서 더 높은 수준의 자동화와 효율성을 제공합니다.

**현재 (2026년 초 기준)**:
- Prompt Engineering: 개인/소규모 팀 표준
- Context Engineering: 스타트업/중소기업 도입 중
- Harness Engineering: 대형 기술 기업 상용화
- Agentic Engineering: 연구/초기 실험 단계

**당신의 선택**: 현재 필요한 수준부터 시작하되, 각 단계의 핵심을 이해하고 점진적으로 업그레이드하는 것이 최선의 전략입니다.
